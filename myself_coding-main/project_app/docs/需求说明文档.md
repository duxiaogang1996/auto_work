## 需求 1：分月返还补赠费业务
**记录时间**：2026-03-25

### 1. 前端需求

- **输入项**：提供一个输入框用于填入 Cookie。
- **输入项**：提供一个多行文本框用于输入多个手机号（每行一个手机号），并可选支持文件上传（CSV/Excel）。
- **展示项**：执行结果需要在前端页面上以表格的形式展示（可复制、可下载结果）。
- **安全要求**：Cookie 仅在本次请求内使用，不落盘、不写入日志、不写入文档。

### 2. 业务流程 (Business Logic)

针对输入的每一个手机号，系统需要执行以下流程：

1. **查询分月返还记录**：判断处理状态是否为“状态不符未返还”，如果是，记录其对应的生效时间（月份）。
2. **查询账单**：针对上一步记录的“未返还月份”，查询这些月份是否有实际扣款。如果该月有扣款，则记录该扣款金额（可能有多个月份）。
3. **查询业务办理记录（防重）**：检查该号码是否已经有过赠费记录。如果有，需要将历史赠费金额统计出来。
4. **查询当前余额 (赠费前)**：记录赠费操作前的账户余额。
5. **执行赠费操作**：
   - 计算应赠费金额 = `未返还月份扣款的金额总和` - `已赠费金额`。
   - 如果应赠费金额 > 0，则调用赠费接口进行充值/赠费。
6. **查询当前余额 (赠费后) 并校验**：再次查询账户余额，计算（赠费后余额 - 赠费前余额），验证差值是否等于实际操作的赠费金额，最终将所有数据状态汇总返回给前端表格展示。

### 3. 使用接口 (API)

根据业务流程，需要以下接口：

1. `API-1`：查询号码分月返还记录接口。
2. `API-2`：查询号码历史账单/扣款明细接口。
3. `API-3`：查询业务办理记录（赠费记录）接口。
4. `API-4`：查询号码当前余额接口。
5. `API-5`：执行赠费/充值操作接口。

### 3.1 接口整理（结构化摘要）

本节用于把抓包信息整理成“可直接写代码”的结构化内容；同时提示仍需补齐的信息。
说明：Cookie 属于敏感信息，不建议在文档长期保存；后续程序将由前端输入并在请求中临时携带。

#### 通用请求约定（建议）

- 鉴权方式：前端输入 `Cookie`（后端仅内存使用，不落盘、不写日志）。
- 共同请求头：`Accept` / `Accept-Encoding` / `Accept-Language` / `Connection` / `Host` / `Referer` / `User-Agent` / `X-Requested-With`。

#### API-1：查询号码分月返还记录（未返还月份来源）

- 方法：GET
- URL：`http://bossweb.jd.com/busi_web/busi/QryPromDetail_Qry`
- Query：`device_number={手机号}`，`tele_type=GSM`
- 关键字段：
  - `args.list[]`
    - `giveStatus`：处理状态（筛选 `状态不符未返还`）
    - `noGiveReason`：未返还原因（不作为筛选条件，仅用于展示/排查）
    - `effDate`：生效时间（用于推算账期月份）
- 输出：`未返还月份账期列表`（将 `effDate` 转为 `YYYYMM`，用于 API-2 的 `date_bill`）

**抓包请求模板（脱敏）**：
```
GET /busi_web/busi/QryPromDetail_Qry?device_number={手机号}&tele_type=GSM HTTP/1.1
Host: bossweb.jd.com
Accept: application/json, text/javascript, */*; q=0.01
X-Requested-With: XMLHttpRequest
Referer: http://bossweb.jd.com/busi_web/busi/QryPromDetail?menu_id=156
Cookie: <前端输入的 Cookie>
```

**抓包响应关键点（字段映射）**：
- `type == "success"`：本次请求成功
- `args.list_pro[]`：套餐/协议相关信息（目前业务不直接依赖，但可用于展示）
- `args.list[]`：返还明细列表（业务核心）
  - `effDate`：生效时间（示例：`2025-09-01 00:00:00`）
  - `giveStatus`：处理状态（示例：`处理成功` / `状态不符未返还` / `未处理(生效时间未到)`）
  - `releaseFee`：分月返还金额（示例：`1.00`，仅用于解释，不等于扣款）

**注意**：
- 你提供的这组示例响应中 `args.list[]` 的 `giveStatus` 均为 `处理成功`，因此这次示例不会产生“未返还月份”。逻辑上应当是筛选出 `状态不符未返还` 的月份再进入 API-2。
- `effDate` 转账期：`YYYY-MM-DD ...` → `YYYYMM`（如 `2025-09-01 00:00:00` → `202509`）。

#### API-2：查询账单扣款（未返还月份扣款金额来源）

- 方法：GET
- URL：`http://bossweb.jd.com/busi_web/busi/queryInfoBill`
- Query：`device_number={手机号}`，`date_bill={YYYYMM}`，`tele_type=GSM`
- 关键字段：
  - `args.queryBillDto.totalDeduct`：当月总扣款金额（字符串金额，如 `89.00`）
  - 可选：`args.queryBillDto.acctItemDtoPage.dataRows[]`：明细（辅助解释扣费项）
- 输出：`未返还月份扣款总和`（对所有未返还月份 `totalDeduct` 求和）

#### API-3：查询业务办理记录（防止重复赠费）

- 方法：GET
- URL：`http://bossweb.jd.com/busi_web/busi/querysubsorderquery`
- Query：`MSISDN={手机号}`，`teleType=GSM`，`startDate={YYYY-MM-DD}`，`endDate={YYYY-MM-DD}`，`pageIndex`，`rowPerPage`，`contactChannel`
- 关键字段：
  - `args.orderDtoList.dataRows[]`
    - `UserEventName` / `UserEventCode`：事件类型（示例：`充值缴费` / `18`）
    - `Comments`：备注（示例：`充值缴费 企业现金2.00元`，可从中提取赠费金额）
    - `OrderStatus`：状态（建议只统计 `竣工`）
- 输出：`已赠费金额总和`（按规则过滤记录后，从 `Comments` 抽取金额并累加）

## 需求 2：自动停机业务
**记录时间**：2026-03-25

### 1. 前端需求

- **输入项**：前端获取需要停机的号码（支持 CSV 文件上传）。
- **展示项**：执行结果需要在前端页面上以表格的形式展示（可复制、可下载结果）。

### 2. 业务流程 (Business Logic)

针对输入的每一个手机号，系统需要执行以下流程：

1. **查询基础信息**：查询号码对应的套餐和当前余额。
2. **逻辑判断**：
   - 比较余额和套餐金额：如果余额小于套餐金额（说明不够扣下一期的月租）。
   - 判断状态：当前号码的状态是否是“在用”。
3. **执行停机**：如果满足上述条件（余额小于套餐且在用），则调用停机接口进行停机。
4. **结果展示**：将对应的结果显示在前端。

### 3. 使用接口 (API)

根据业务流程，预期需要以下接口：

1. `API-4`（复用）：查询号码当前余额与基础信息接口（获取 `acct_Balance`，`subs_Plan_Name`，`user_status`）。
2. `API-6`：执行停机接口。

### 3.1 停机相关接口整理

#### API-6：执行停机接口（单号）

- 方法：POST（form）
- URL：`http://bossweb.jd.com/busi_web/busi/dealOpenClose`
- Referer：`http://bossweb.jd.com/busi_web/busi/openCloseJf?para=20&?menu_id=77`
- Form：
  - `misidn={手机号}`
  - `reasonCode={原因码}`（联通停机示例：`90`；电信停机示例：`80`）
  - `tele_type={电信类型}`（示例：`GSM`）
  - `remark={备注}`（可空；示例：`企业方停机-不满下月月租停机`）
- 响应关键字段：
  - `type`：`success` 表示停机成功
  - `content`：结果说明

### 3.2 确认规则记录（需求 2）

1. **自动化程度**：采用 **方案 B**。程序在筛选出“需停机”的号码后，自动调用 API-6 逐个执行停机，并在结果中展示停机请求是否成功（`type=success`）。
2. **套餐金额映射规则**：**自动从套餐名称中提取数字**。例如：从 `"1元年卡【联通制式】"` 中提取出 `1` 作为套餐金额，用于和当前余额进行比较。
3. **"在用"状态枚举值**：API-4 查询返回的 `user_status` 字段，代表"在用"的字符串确认为 `"在用"`。

## 需求 3：配置套餐包、产品包 (SQL 自动生成)
**记录时间**：2026-03-25

### 1. 前端需求

- **输入项**：提供一个分步骤的表单页面（Multi-step form）。
- **交互逻辑**：前端依据所有相关的表结构关系，引导用户分步骤填写对应的信息。
- **展示项**：填写完成后，前端（或后端配合）自动生成对应的 `INSERT` SQL 语句，并在页面上展示（提供"一键复制"或"下载为 .sql 文件"的快捷功能）。

### 2. 业务流程 (Business Logic)

1. **表单数据收集**：用户在前端按步骤填写配置套餐包和产品包所需的各类参数。
2. **关系映射与转换**：系统将前端填写的数据，按照底层数据库的表结构和外键/关联关系，映射到具体的字段上。
3. **SQL 生成**：根据映射好的数据，生成一系列标准的 `INSERT INTO` 语句（需注意字符串转义、日期格式等）。
4. **结果输出**：将生成的 SQL 语句输出给用户，以便用户直接复制到数据库客户端（如 Navicat/DBeaver）中执行。

### 3. 使用接口 (API)

- 本需求主要为内部工具逻辑，不涉及外部业务 API 的调用。
- 需要在本地系统内部提供一个 `/api/sql/generate` (暂定) 接口，接收前端提交的 JSON 表单数据，返回生成的 SQL 字符串。

## 需求 8：两种类型余额查询
**记录时间**：2026-04-10

### 1. 前端需求

- **输入项**：复用现有输入框，支持 Cookie 和多个手机号输入（每行一个，也支持文件上传）。
- **查询类型选择**：前端提供两种查询模式选择：
  - **模式 1**：只查询状态和余额
  - **模式 2**：查询余额和具体的账本明细
- **账期参数**：查询账本明细时需要指定账期（YYYYMM 格式）。
- **展示项**：
  - 模式 1 表格展示：手机号、套餐、状态、余额、激活时间
  - 模式 2 表格展示：手机号、套餐、状态、激活时间、余额、欠费、账本类型、项目名称、金额、单位、生效日期、失效日期
- **导出支持**：支持结果导出为 Excel。
- **安全要求**：Cookie 仅在本次请求内使用，不落盘、不写入日志、不写入文档。

### 2. 业务流程 (Business Logic)

**两种查询逻辑**：

#### 模式 1：只查询状态和余额

1. **第一步**：先调用**综合查询接口**（API-4 querySubsBaseInfo），这个接口可以查询状态和余额。
2. **判断结果**：
   - 如果综合查询**成功**：直接返回结果（手机号、套餐、状态、余额、激活时间）。
   - 如果综合查询**查询不到**（可能是号码已销户）：
     - **第二步**：需要再调用**余额查询接口**（API-11 queryBalances）。
       - 如果余额查询接口**可以查到**：说明号码状态为**已销户**，使用余额查询接口返回的余额信息。
       - 如果余额查询接口**也查不到**：说明号码为**未开户**。

#### 模式 2：查询状态和具体的账本明细

1. **第一步**：调用**综合查询接口**（API-4）查询状态、套餐、余额、激活时间。
2. **第二步**：调用**余额查询接口**（API-11 queryBalances）查询具体的账本明细（金额账本和资源账本）。
3. **输出**：将基础信息和账本明细合并后返回，每个账本明细一行展示。

### 3. 使用接口 (API)

根据业务流程，需要以下接口：

1. `API-4`（复用）：综合查询接口 `querySubsBaseInfo` - 获取号码基础信息（套餐、状态、余额、激活时间）。
2. `API-11`（新增封装）：余额查询接口 `queryBalances` - 查询余额和账本明细。

### 3.1 接口整理（结构化摘要）

#### API-11：余额查询接口（查询余额和账本明细）

    - `effDate`：生效日期
